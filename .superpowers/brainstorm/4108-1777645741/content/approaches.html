3 Architectural Approaches

How should OIDC authentication flow between React, Quarkus BE, and Keycloak?

A

SPA OIDC — React manages tokens (PKCE)

React handles the full OIDC Authorization Code + PKCE flow directly with Keycloak. After login, the FE stores the access token and sends it as a Bearer header on every API call. Quarkus BE validates the JWT on each request using quarkus-oidc (stateless).

Browser → Keycloak (PKCE login)
Browser ← access_token + id_token
Browser → Quarkus BE (Bearer token)
Quarkus BE → MySQL (reads)
Quarkus BE → Kestra API (retrigger)

Pros

  • Standard React SPA pattern
  • Stateless BE — easy to scale
  • Simple Quarkus config (just JWT validation)
  • Well-documented with Keycloak

Cons

  • Token lives in browser memory/storage
  • FE must handle token refresh

⭐ Recommended

B

BFF — Quarkus BE owns the OIDC session

Quarkus BE is the registered OIDC client. The browser redirects to Keycloak via Quarkus, which handles the Authorization Code Flow server-side and issues a session cookie to React. Tokens never reach the browser.

Browser → Quarkus BE → Keycloak (login)
Quarkus BE ← access_token (stored server-side)
Browser ← session cookie
Browser → Quarkus BE (cookie)
Quarkus BE → MySQL (reads)
Quarkus BE → Kestra API (retrigger)

Pros

  • Tokens never exposed to browser
  • More secure — no XSS token theft risk
  • BE controls session lifecycle

Cons

  • Stateful BE sessions (fine for 5–20 users)
  • More complex Quarkus OIDC config
  • CORS + cookie handling trickier in dev
C

Hybrid — React login, Quarkus token exchange

React logs in via Keycloak PKCE (same as A), but sends only the id_token to Quarkus BE. Quarkus performs a token exchange with Keycloak to get a service-level access token for calling Kestra API. User identity comes from the validated id_token.

Browser → Keycloak (PKCE login)
Browser ← id_token + access_token
Browser → Quarkus BE (id_token)
Quarkus BE → Keycloak (token exchange)
Quarkus BE → MySQL + Kestra API

Pros

  • Service account separation — Kestra sees Quarkus, not the user
  • Fine-grained scopes per service

Cons

  • Token exchange requires Keycloak Enterprise or specific config
  • Most complex of the three
  • Overkill for an internal tool with 20 users
